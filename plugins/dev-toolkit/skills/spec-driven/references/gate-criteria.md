# ゲート定義と通過条件（spec-driven 用）

ゲート判定（SKILL.md Step 2〜3）で使う。**各ゲートは「揃っていないものを可視化して先に進めない」ための関所**。通過条件は observable（誰が見ても判定が一致する形）で確認する。

## 5ゲートの全体像

| ゲート | 目的 | 主な成果物 | 最頻の違反 |
|---|---|---|---|
| G0 要件 | 何を作るかの合意 | 要求文・受け入れ基準・スコープ外 | 曖昧なまま実装に着手 |
| G1 仕様 | どこに何を書くかの確定 | 仕様文書の更新（ドキュメント先）・影響度評価 | コードだけ変えて仕様を放置 |
| G2 計画 | 実装可能な単位への分解 | タスク分解・テスト計画 | 大きすぎる一括実装 |
| G3 実装 | 仕様に沿った実装 | コード + テスト | 仕様にない挙動の追加 |
| G4 検証 | 完了の証明 | 受け入れ基準チェック結果・乖離ゼロ確認 | 「動いたから完了」 |

## G0 要件: 通過条件

- [ ] 要求が1〜2文で言語化されている（Why / What）
- [ ] **受け入れ基準が 3〜5 個**、観測可能な形（実行して判定できる）で書かれている
- [ ] スコープ外（今回やらないこと）が明記されている
- [ ] 要求の曖昧点は次のいずれかの状態になっている（勝手に補完しない）: (a) 質問して解消済み (b)「前提」として明記 (c) **バッチ実行等で対話できない場合**、SD-n に質問を記録し、回答なしで安全に進められる範囲にスコープを限定済み

曖昧な要求（「いい感じに」「使いやすく」）は、**実装案ではなく質問リスト**を返すのが G0 の正しい出力。

## G1 仕様: 通過条件

- [ ] 変更が影響する仕様文書を判断マトリクスで特定した（→ spec-docs-map.md）
- [ ] **仕様文書を先に更新した**、または依頼範囲が G2 以前のため**更新案（差分）を提示した**〔この場合の状態は「提案通過」〕。文書が存在しない場合は変更が影響する文書のみ最小構成で新設（`${CLAUDE_PLUGIN_ROOT}/docs-template/` の対応テンプレートを使用。frontmatter は組み込み済み）
- [ ] 影響度を LOW / MEDIUM / HIGH で評価し、HIGH なら関係者確認・ADR・移行計画の要否を判定した
- [ ] 仕様の更新内容が受け入れ基準と矛盾しない

## G2 計画: 通過条件

- [ ] 実装を検証可能な単位に分解した（1 単位 = 1 つの明確な出力。目安 1〜3 日、セッション内なら 5〜15 分粒度）
- [ ] 各単位に完了条件（テスト・確認方法）がある
- [ ] 分解した単位の合計が受け入れ基準を全部カバーしている（対応表で確認）
- [ ] 触ってよい範囲・触らない範囲（ファイル・モジュール）を明示した

## G3 実装: 通過条件

- [ ] 実装は G1 の仕様を参照して行った（仕様にない挙動を足していない。必要になったら G1 に戻って仕様を先に更新）
- [ ] 各単位のテストが存在し、パスしている
- [ ] 既存テストが壊れていない
- [ ] スコープ外に踏み出していない（踏み出しが必要なら G0 に戻すか別タスク化）

## G4 検証: 通過条件

- [ ] **受け入れ基準を1つずつ実行して判定した**（チェック結果を記録。未達があれば完了と言わない）
- [ ] 仕様文書と実装の乖離がない（変更が仕様に反映済みか最終確認）
- [ ] 影響度 MEDIUM 以上の場合、関連文書（テスト・運用）も更新した
- [ ] 学び（ハマりどころ・判断）があれば記録した（ナレッジ蓄積へ）

## ゲート運用の原則

- **飛ばさない・戻ってよい**: ゲートは順に通る。ただし後段で前提が崩れたら前のゲートに戻るのは正常（戻った事実と理由を進行表に記録する）
- **通過条件は自己申告にしない**: チェック項目は「実行した証拠」（質問と回答・更新した文書のパス・テスト結果）とセットで記録する
- **止まる勇気**: 通過条件が満たせないとき、無理に進めず「何が足りないか」を報告して指示を仰ぐのが正しい挙動
- 軽微なタスク（typo 修正等）は G0 で「軽量モード」を宣言し、G1〜G2 を簡略化してよい（省略ではなく簡略化。受け入れ基準1個と影響度 LOW の確認は必須）
