# ハーネス設計アンチパターンカタログ（harness-review 用）

指摘の裏付け（SKILL.md Step 3）で使う。**検出サイン → 何が起きるか → 推奨対応**の形で参照し、指摘文に流用する。

## 構成系

### 過剰マルチエージェント
- **検出サイン**: 「最先端だから」「柔軟性のため」という理由の複数体構成。関心事が1〜2個なのにフル・オーケストレーション。同期ポイントが頻発する設計
- **何が起きるか**: つなぎ目ごとのエラー累積、調整コストが並列化の利得を食い潰す、デバッグ困難
- **推奨対応**: タスク構造で正当化できない部分を単一エージェント/固定フローに畳む。採否を ADR に記録

### 固定手順の LLM 化
- **検出サイン**: 入出力が決まった変換・集計・整形を毎回 LLM にやらせている
- **何が起きるか**: 非決定的な結果・コスト増・評価不能
- **推奨対応**: 決定的処理として実装し、判断が要る部分だけ AI に残す

### 無限改善ループ
- **検出サイン**: 生成→評価→修正のループに上限がない
- **何が起きるか**: コスト暴走・収束しない改善
- **推奨対応**: 回転数上限（目安3回）と打ち切り時の扱いを明記

## 境界系

### Provider 散在（ベンダー直結）
- **検出サイン**: LLM API・認証・監視 SDK の呼び出しが複数モジュールに直接書かれている
- **何が起きるか**: モデル変更・障害時に全箇所修正、テスト不能、ロックイン
- **推奨対応**: interface で抽象化し実装を1箇所に集約。テスト用 fake を用意

### 境界の侵食
- **検出サイン**: 「急ぎの修正」でレイヤーを跨いだ参照が増えている。環境変数の直接参照が散在
- **何が起きるか**: 変更影響が読めなくなり、AI にも人間にも危険な codebase になる
- **推奨対応**: 依存方向ルールの明文化 + 静的解析での機械的検出（人間の注意力に頼らない）

## コンテキスト系

### 全部渡し（Context Bomb）
- **検出サイン**: エージェントに全ファイル・全履歴・無制限のログを渡す設計。動的に取得するデータに件数・サイズ制限がない
- **何が起きるか**: 精度低下（中間忘却）・コスト増・本番データ量で爆発
- **推奨対応**: 必要最小限の選別基準とフィルタ・上限を設計に明記

### つなぎ目の未設計
- **検出サイン**: エージェント間の受け渡しが「前の出力をそのまま次へ」だけ
- **何が起きるか**: 下流の精度が上流のノイズに引きずられる
- **推奨対応**: 方向性の指定・不要情報の削除・フォーマット指定を接続ごとに定義

## 安全系

### 停止不能設計
- **検出サイン**: 停止条件・停止権限・kill switch の記載がない。自律ループに人間の介入点がない
- **何が起きるか**: 暴走時に止められない（Critical）
- **推奨対応**: 停止条件（指標・イベント）と停止権限、復旧手順を最優先で追加

### 無限リトライ / タイムアウトなし
- **検出サイン**: 「失敗したら再試行」とだけ書かれている。外部 API 呼び出しに時間上限がない
- **何が起きるか**: ハング・コスト暴走・障害の全体伝播（Critical〜Warning）
- **推奨対応**: 回数上限+バックオフ、ステップ別タイムアウト、フォールバック先

### 全権限エージェント
- **検出サイン**: 利用者より広い権限、削除・送信を含む実行権限の無条件付与、承認ゲートなしの破壊的操作
- **何が起きるか**: 誤実行・漏洩が構造的に防げない（Critical）
- **推奨対応**: 最小権限・承認ゲート・監査ログの3点セット

## 運用系

### 自己申告完了（Silent Failure）
- **検出サイン**: 完了判定がエージェントの「完了しました」だけ。成果物の存在・形式検証がない
- **何が起きるか**: 失敗の見逃しが下流に伝播
- **推奨対応**: 完了時検証をゲート化（存在・形式・テスト）。検証不能なら人間確認に回す

### 観測ゼロ運用
- **検出サイン**: ログ・トレース・KPI の設計が存在せず言及もないまま運用を開始する設計。品質低下に気づく手段がない（「別途検討」等と未定が明記されている場合は指摘でなく情報不足〔HR〕として扱い、本番化前の解消を確認事項にする）
- **何が起きるか**: 障害の原因追跡不能、劣化の静かな進行
- **推奨対応**: 最低限の判断ログ（入力・参照・出力・承認）と回帰テストの運用組み込み

### 根拠なきルール
- **検出サイン**: 制約・規約に理由の記録がない。「なぜか分からないが守っているルール」
- **何が起きるか**: 黙って例外が増え、ルールが形骸化
- **推奨対応**: ADR とルールの双方向リンク、Superseded 運用

## 良い設計のサイン（誤検知防止）

以下は**指摘対象ではない**。見つけたら「良い点」として記録する:

- 単一エージェント+固定フローで足りる構成をそのまま単純に保っている
- 段階導入（承認付き→実績を見て自動化拡大）の設計。**ただし拡大段階で承認ゲートを全廃する計画はこのサインに該当しない**（安全系の観点で判定する）
- 影響の小さい操作から権限を絞って始める設計
- 「やらないこと」が明記されている
- 失敗ケースが先に列挙され、対応が決まっている
