# ハーネスレビューの観点と重大度基準（harness-review 用）

レビュー実施（SKILL.md Step 2〜3）で使う。**7観点を順に検査し、判定は OK / NG / 情報不足 / 対象外 の4値**で記録する。推測で NG にしない（情報不足は情報不足として指摘する）。

## 観点1: 構成の妥当性（過剰/過少設計）

- [ ] マルチエージェント構成の正当性: 並列の時間短縮 > 分離・同期コストか。独立した視点が品質に効くか。**「複雑そうだから複数」は過剰設計のサイン**
- [ ] タスク規模と構成の釣り合い（関心事1つ=単一 / 2つ=2〜3体 / 独立3つ以上=フル構成）
- [ ] 決定的処理・AI 判断・エージェントの割り付け: 決められる処理を決め打ちにしているか（固定手順を LLM にやらせていないか）
- [ ] 接続点（つなぎ目）の数: 直列3〜4工程・並列3系統・改善ループ3回転を大きく超える場合、エラー累積の検討があるか

## 観点2: 境界と依存

- [ ] 依存方向が一方向に固定されているか（内側が外側を知らない。逆依存・層飛ばしの禁止）
- [ ] 外部サービス（認証・監視・LLM API・外部 SDK）が Provider 境界で抽象化されているか（直接呼び出しの散在はベンダーロックイン+障害伝播）
- [ ] 外部入力が境界で検証・型付けされているか（Parse, don't validate）
- [ ] 設定・環境変数の参照が1箇所に集約され、起動時に検証されるか

## 観点3: コンテキスト管理

- [ ] 各エージェント/ステップに渡すコンテキストが必要最小限か（全部渡しは精度低下+コスト増）
- [ ] タスク分割の粒度（1単位=1つの明確な出力）
- [ ] つなぎ目の設計（次工程への方向性・不要情報の削除・フォーマット指定）があるか
- [ ] 長い作業でのコンテキスト圧迫への対策（分割・要約・状態ファイル）があるか

## 観点4: ツール・権限設計

- [ ] **最小権限**: エージェントの参照・実行権限が利用者の権限を超えないか。実行権限（書き込み・送信・削除）が必要最小か
- [ ] 破壊的操作に承認ゲートがあるか
- [ ] サンドボックス・作業領域の分離（並列時の書き込みスコープ非重複）
- [ ] 認証情報の扱い（ハードコード・ログ出力の禁止）

## 観点5: 失敗設計

- [ ] リトライに上限とバックオフがあるか（**無限リトライは Critical**）
- [ ] 各ステップにタイムアウトがあるか（外部依存の遅延で全体が止まらないか）
- [ ] フォールバック（AI が続行不能時の代替経路）があるか
- [ ] **人間へ戻す条件**（低確信・重大カテゴリ・リトライ上限）が明文化されているか
- [ ] **停止条件と停止権限**（何が起きたら止めるか・誰が止められるか）があるか（無いのは Critical）
- [ ] エスカレーションの階段（リトライ→別アプローチ→分割→人間）が設計されているか

## 観点6: 状態管理・観測性

- [ ] 進行状態が永続化され、中断から復帰できるか（長時間・マルチステップ構成の場合）
- [ ] ログ・トレースの設計（何を・どこに・どの期間。判断根拠が追えるか）
- [ ] 完了時検証がゲート化されているか（成果物の存在・形式チェック。自己申告の「完了しました」で進まないか）
- [ ] 品質・回帰の計測（評価データセット・回帰テスト・KPI）が運用に組み込まれているか

## 観点7: ガバナンス・保守性

- [ ] 設計判断が記録されているか（ADR: 背景・選択肢・根拠・見直し条件）
- [ ] ルール・制約に根拠へのリンクがあるか（「なぜあるか不明なルール」の予防）
- [ ] 3層防御（予防=ガイドライン / 即時=型・lint / 検出=CI ゲート）のどこで何を守るかが決まっているか
- [ ] エージェントへの指示資産（プロンプト・スキル）が版管理されているか

## 重大度の判定基準

| 重大度 | 基準 | 例 |
|---|---|---|
| Critical | 事故（暴走・データ破壊・漏洩・無限消費）に直結する欠陥 | 停止条件なし・無限リトライ・全権限付与・破壊的操作の無承認自動実行 |
| Warning | 運用で高確率で問題化する設計不備 | フォールバックなし・ログ設計なし・Provider 境界なし・完了検証なし |
| Suggestion | 改善で品質・保守性が上がるもの | つなぎ目設計の明文化・ADR の充実・粒度の調整 |

- 重大度は**影響と発生確率**で決める。好みの問題（命名・書式）は指摘しない
- **NG には必ず根拠（どの観点のどの項目か）と推奨対応を付ける**
- 設計書に書かれていないことは「未実装」と断定せず「情報不足」として確認事項にする
