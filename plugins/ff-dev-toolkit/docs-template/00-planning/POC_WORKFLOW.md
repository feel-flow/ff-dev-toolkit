# PoC ワークフロー

## 目的

PoC（Proof of Concept）は、実装前に不確実性を小さくするための短期検証である。完成品を作る活動ではなく、仕様化・ADR作成・追加検証・中止のいずれかを判断するために実施する。

このガイドでは、PoCの開始条件、検証項目の絞り方、`poc/` ディレクトリの扱い、結果記録テンプレート、PoCで得た知見を仕様文書へ反映する方法を定義する。

## PoC を開始する条件

PoCは、次のいずれかに該当する場合に実施する。

- 技術選定や外部サービス連携に未検証の前提がある
- パフォーマンス、セキュリティ、運用制約など、設計だけでは判断できないリスクがある
- 仕様化前にユーザー価値や業務フローの成立性を短時間で確認したい
- 本実装に入ると手戻りが大きい判断が残っている

PoCを開始する前に、検証項目、終了条件、成果物の保存先を明記する。検証項目が多い場合は、PoCではなく設計タスクまたは複数Issueへ分割する。

## 検証項目チェックリスト

PoCの検証項目は、必ず1〜3項目に絞る。

```markdown
## PoC 検証項目

- [ ] 検証項目1: [最も重要な不確実性]
- [ ] 検証項目2: [必要な場合のみ]
- [ ] 検証項目3: [必要な場合のみ]

## 検証しないこと

- [ ] [今回のPoCでは扱わない範囲]
```

検証項目を書くときは、次の形式を推奨する。

- 「できるか」ではなく「どの条件なら成立するか」を書く
- 成功条件と失敗条件を観察可能な形で書く
- 本実装品質、網羅テスト、UI仕上げなど、PoCの目的外の作業を明示的に除外する

## `poc/` ディレクトリ分離

PoCのコード、メモ、実験データは本実装と分離する。推奨配置は次の通り。

```text
poc/
├── README.md
├── YYYY-MM-DD-topic/
│   ├── README.md
│   ├── src/
│   ├── notes.md
│   └── result.md
└── .gitignore
```

### 管理方針

| 対象                        | Git管理              | 理由                                     |
| --------------------------- | -------------------- | ---------------------------------------- |
| `README.md`                 | 管理する             | PoCの目的、実行方法、結果への導線を残す  |
| `result.md`                 | 管理する             | 意思決定の根拠として再利用する           |
| 小さなサンプルコード        | 必要に応じて管理する | 仕様化やADRの根拠になる場合のみ残す      |
| 生成物、ログ、大容量データ  | 管理しない           | 再生成可能または機密・肥大化リスクがある |
| APIキー、認証情報、個人情報 | 管理しない           | セキュリティ上の禁止事項                 |

### `.gitignore` 推奨設定

PoCごとに必要最小限を調整する。機密情報や大容量ファイルは必ず除外する。

```gitignore
# PoC local artifacts
poc/**/.env
poc/**/.env.*
poc/**/node_modules/
poc/**/dist/
poc/**/build/
poc/**/coverage/
poc/**/.cache/
poc/**/tmp/
poc/**/*.log
poc/**/*.sqlite
poc/**/*.db
poc/**/*.csv
poc/**/*.parquet
```

PoCコードを本実装へ流用する場合は、`poc/` からコピーせず、設計・実装規約に合わせて正式な配置先へ作り直す。PoC由来の制約や判断は、仕様文書またはADRに残す。

## PoC の終了条件

PoCは、次のいずれかで終了する。

| 結果       | 判定条件                                                 | 次のアクション                    |
| ---------- | -------------------------------------------------------- | --------------------------------- |
| 成功       | 検証項目が満たされ、本実装に進む判断材料が揃った         | 仕様化、ADR作成、実装Issue化      |
| 失敗       | 成立条件を満たせない、または制約が受け入れ不可と判明した | 代替案検討、中止、スコープ再定義  |
| 要追加検証 | 判断に必要な情報が不足している                           | 検証項目を絞り直し、追加PoCを作成 |

終了時は必ず `result.md` を作成し、検証結果と次のアクションを明記する。PoCを終了できない場合は、検証項目が大きすぎる可能性が高いため、Issueまたは検証観点を分割する。

## PoC 結果記録テンプレート

```markdown
## PoC: [検証タイトル]

- 検証項目:
- 検証結果: 成功 / 失敗 / 要追加検証
- 発見事項:
- 制約・制限:
- 次のアクション: 仕様化 / ADR作成 / 追加PoC / 中止

## 前提

- [検証時に置いた前提、環境、入力データ]

## 実施内容

1. [実施した手順]
2. [観察した事実]

## 判断

- 成功と判断した理由:
- 失敗または追加検証が必要と判断した理由:

## 反映先

- [ ] 01-context/PROJECT.md
- [ ] 02-design/ARCHITECTURE.md
- [ ] 02-design/DOMAIN.md
- [ ] 03-implementation/PATTERNS.md
- [ ] 04-quality/TESTING.md
- [ ] 05-operations/DEPLOYMENT.md
- [ ] 06-reference/DECISIONS.md
```

テンプレートには観察した事実と判断を分けて書く。推測、期待、未確認事項は「発見事項」ではなく「制約・制限」または「次のアクション」に記録する。

## PoC から仕様へのマッピング

PoCで得た知見は、次の表に従ってSSOTへ反映する。

| PoCで得た知見              | 反映先文書                                            | 反映内容                               |
| -------------------------- | ----------------------------------------------------- | -------------------------------------- |
| 技術的な制約・選択         | [ARCHITECTURE.md](../02-design/ARCHITECTURE.md) + ADR | 採用技術、非採用理由、代替案、制約     |
| ビジネスルールの発見       | [DOMAIN.md](../02-design/DOMAIN.md)                   | ルール、例外、用語、状態遷移           |
| 実装パターンの確立         | [PATTERNS.md](../03-implementation/PATTERNS.md)       | 実装方針、命名、エラー処理、境界の扱い |
| テスト戦略の知見           | [TESTING.md](../04-quality/TESTING.md)                | テスト観点、モック方針、品質ゲート     |
| 運用上の制約               | [DEPLOYMENT.md](../05-operations/DEPLOYMENT.md)       | 環境変数、監視、デプロイ、ロールバック |
| プロダクト価値や要件の変更 | [PROJECT.md](../01-context/PROJECT.md)                | 目的、対象ユーザー、受け入れ条件       |
| 意思決定の履歴             | [DECISIONS.md](../06-reference/DECISIONS.md)          | ADR、採用・却下の理由、判断日          |

### 反映手順

1. `result.md` の「発見事項」と「制約・制限」を読み、事実と判断を分ける。
2. 上のマッピング表でSSOTを決める。
3. 反映先文書を更新し、PoC結果への参照を残す。
4. 技術選定や不可逆な判断が含まれる場合はADRを追加する。
5. 反映後、PoC IssueまたはPRに「反映先」を記録する。

## レビューチェックリスト

PoC完了時のレビューでは、次を確認する。

- [ ] 検証項目は1〜3項目に絞られている
- [ ] 成功 / 失敗 / 要追加検証のいずれかで終了している
- [ ] 観察した事実と判断が分かれている
- [ ] `poc/` の生成物、ログ、機密情報がGit管理対象外になっている
- [ ] PoCの知見が適切な仕様文書またはADRへ反映されている
- [ ] 本実装へ進む場合、PoCコードをそのまま流用せず正式な配置・品質基準で作り直す方針になっている
