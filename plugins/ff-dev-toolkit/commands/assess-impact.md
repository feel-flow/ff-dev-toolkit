# /assess-impact — 変更影響度評価

変更対象の文書・コードを分析し、影響度を LOW / MEDIUM / HIGH で評価します。HIGH の場合は ADR・移行計画の要否判定を促します。

## 前提

- 変更対象のファイルまたは変更内容が明確であること
- プロジェクトに `docs/` 配下のコア文書（最小構成: MASTER・PROJECT・ARCHITECTURE の3文書）が存在すること（`/init-docs` で初期化可能）

## 手順

### 1. 変更対象の特定

以下のいずれかの方法で変更対象を特定します:

- **ファイル指定**: ユーザーが変更対象ファイルを指定
- **git diff**: 現在の未コミット変更から自動検出
- **Issue 参照**: Issue 番号から変更予定を推定

### 2. 影響度の評価

変更を次の3分類のいずれかに割り当て、影響度を判定します。**影響度は「何に影響するか」（波及範囲）で測ります。行数・文字数・ファイル数などの変更量で判定してはいけません。**

| 分類 | 影響度 | 基準 | 例 |
| ---- | ------ | ---- | -- |
| **文言修正** | **LOW** | 意味を変えない表現・体裁の修正で、他の箇所へ波及しない | typo修正、コメント調整、言い回しの変更 |
| **概念追加** | **MEDIUM** | 既存の枠組みを保ったまま要素を足す。他の文書・コードへ波及するが、既存設計は維持できる | フィールド追加、新規関数追加、既存APIにオプショナルパラメータ追加 |
| **概念再定義** | **HIGH** | 既存の枠組みそのものを変える・削除する。既存設計を維持できない | DB スキーマ変更、認証方式変更、API破壊的変更、機能削除 |

境界は次の規則で判定します:

- **LOW/MEDIUM の境界 = 波及の有無**: 変更が他の箇所に波及するかで判定する。文言修正でも、その文言をコードやツールが解釈している場合は波及があるため MEDIUM とする
- **MEDIUM/HIGH の境界 = 既存設計の維持可能性**: 既存の枠内での拡張は MEDIUM、枠組みの再設計を要するものは HIGH。ロールバックが困難な変更（データ移行を伴う等）は HIGH とする
- **変更量では判定しない**: 単一ファイル1行の変更でも波及すれば MEDIUM 以上（例: デフォルト値の変更が全利用者の挙動を変える場合は HIGH になり得る）。複数ファイルにまたがる変更でも、波及のない機械的な文言修正なら LOW
- **複数の変更単位が混在する場合**: 独立した変更単位ごとに分類・影響度を評価し、**全体の影響度は最も高いものを採る**（低い変更に紛れた高影響変更を見落とさない）

### 3. 影響を受ける文書のリストアップ

変更内容に基づいて、更新が必要な文書を特定します:

- **MASTER.md**: プロジェクト全体の構成が変わる場合
- **PROJECT.md**: ビジョン・対象ユーザー・主要機能・非機能要件・スコープが変わる場合
- **ARCHITECTURE.md**: システム構成・技術選定が変わる場合
- **DOMAIN.md**: ビジネスルール・エンティティが変わる場合
- **PATTERNS.md**: 実装パターン・規約が変わる場合
- **TESTING.md**: テスト戦略が変わる場合
- **DEPLOYMENT.md**: デプロイ・運用手順が変わる場合
- **DECISIONS.md**: 技術的判断が含まれる場合

### 4. 結果の出力

以下の形式で結果を出力してください:

```markdown
## 変更影響度評価

### 変更概要
[変更内容の簡潔な説明]

### 分類: [文言修正 / 概念追加 / 概念再定義]

### 影響度: [LOW / MEDIUM / HIGH]

（複数の変更単位が混在する場合は、単位ごとの分類・影響度を列挙したうえで、最も高い影響度を全体の影響度とする）

### 判定理由
- [波及の有無（LOW/MEDIUM境界）・既存設計の維持可能性（MEDIUM/HIGH境界）に基づく理由]
- [理由2]

### 影響を受ける文書
| 文書 | 影響内容 | 更新必須？ |
| ---- | -------- | ---------- |
| ARCHITECTURE.md | [具体的な影響] | はい/いいえ |
| PATTERNS.md | [具体的な影響] | はい/いいえ |

### HIGH の場合のみ: 必須手順の判定結果
- **影響範囲**: [影響を受ける文書・コード・外部システムの列挙]
- **ADR**: [要 / 不要] — [理由]（要の場合は DECISIONS.md へ作成: コンテキスト・決定・却下した代替案・影響の4点）
- **移行計画**: [要 / 不要] — [理由]（要の場合は Phase 1→2→3 で策定）
- **ロールバック計画（推奨）**: [策定する / 見送る] — [理由]

### 推奨アクション
1. [アクション1]
2. [アクション2]
```

### 5. HIGH 判定時の追加アクション

影響度が HIGH の場合、**必須手順を済ませるまで実装に着手せず**、以下を実施してください:

- **影響範囲の洗い出し（必須）**: 影響を受けるコア文書・コード・外部システムを漏れなく列挙する
- **ADR の要否判定（必須）**: 意思決定の記録として ADR 作成の要否を判定し、必要と判定した場合は DECISIONS.md に作成する。ADR にはコンテキスト・決定・却下した代替案・影響の4点を含める
- **移行計画の要否判定（必須）**: 後方互換性が失われる変更など、必要と判定した場合は Phase 1/2/3 の段階的移行計画を策定する
- **段階的リリース・ロールバック計画（推奨）**: 段階的にリリースし、各段階でロールバックできる状態を保つこと、ロールバックのトリガー条件と手順をあらかじめ明記することを推奨する
- **レビュー範囲の拡大（推奨）**: 影響を受けるすべての文書のレビューを推奨する

要否判定の結果は、必要・不要のいずれでも理由とともに評価結果へ明記してください。

```
🚨 HIGH 影響度 — 実装着手前に以下を完了してください

1. 影響範囲（文書・コード・外部システム）を洗い出してください
2. ADR の要否を判定してください（必要な場合は DECISIONS.md のテンプレートで作成）
3. 移行計画の要否を判定してください（必要な場合は Phase 1→2→3 で策定）
4. 【推奨】ロールバックのトリガー条件と手順を明記してください
5. 【推奨】影響を受ける文書のレビューを実施してください
```

## 重要ルール

- 影響度は波及範囲で判定すること。行数・文字数・ファイル数などの変更量で判定しないこと
- 影響度の判定は保守的に行うこと（迷ったら、より高い影響度として扱う）
- 影響を受ける文書は漏れなくリストアップすること
- HIGH 判定時は ADR・移行計画の要否判定を必ず実施し、不要と判定した場合も理由を記録すること
- 変更内容が不明確な場合は、ユーザーに確認を求めること
